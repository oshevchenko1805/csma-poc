## Что сделано в этой сессии (всё в main)

1. **`AttackContext.monitors` + `MonitorHandle` Protocol** (`attacks/base.py`) —
   attack-слой видит живые мониторы, не импортируя `runners/`.
2. **`Monitor.disable_local_detectors()`** — глушит локальные детекторы,
   монитор остаётся жив (listener/tick/peer-pub работают).
3. **`DetectorTakeoutInjector`** — главный сценарий (см. RESULTS_NOTES R1).
4. **`MonitorTakeoutInjector`** с параметром `takeout_uav` — SPOF-сценарий (R2).
5. **`SequentialAttackInjector`** (`attacks/composite.py`) — композиция атак
   без правки ExperimentRunner (fire по порядку, cleanup в обратном).
6. **`TrajectoryRecorder`** (`runners/trajectory.py`) — внешний ground truth
   из Gazebo. Вкручен в ExperimentRunner + run_one, живо проверен.
7. **Фикс аналайзера**: `impact_scope` теперь с тем же временным якорем, что
   и `detected` (R5).
8. Регистрация сценариев в `run_one` / `run_batch` / `report`.
9. **Данные:** `runs_v2/` — 30 прогонов detector_takeout (batch N=10 × 3 арх).

**Результаты — все в `RESULTS_NOTES.md`. Читать его первым.**
Кратко: detection A 0/10, B 0/10, **C 10/10** (mesh-only, 20 cross_check,
0 локальных); A≠B впервые (SPOF); mesh платит 3× латентностью;
физический снос 50 м измерен, **loiter его не остановил**.

## Решения, которые НЕ переобсуждать (обоснованы, проверены)

- **Работу не переделывать.** Прогалина (2.9) — интеграция
  detection+isolation+recovery, а не детекция. Уход в ML = другая работа и
  потеря своей ниши. Вклад = архитектурный паттерн + его эмпирическая
  характеризация (design science).
- **Детектор — не вклад.** Он сменный компонент через DI. Это надо явно
  написать в Ch.4, а не ждать вопроса рецензента.
- **detector_takeout оставляет монитор живым** — иначе глушится peer-pub и
  cross_check слепнет, C деградирует до B по механической причине (R1).
- **monitor_takeout бьёт по соседу, не по цели** — иначе 0%/0%/0% (R2).
- **impact_scope считает только post-attack события** — pre-attack не может
  быть следствием атаки (R5).
- **Траектория пишется только при `--mission mavsdk`**, исключена из
  `merged.jsonl`, рекордер не может уронить полёт.
- **CPU/RAM не мерить** — в VM под 3×PX4 это шум.
- **Ground truth только из Gazebo.** Оценка PX4 отравлена спуфом по
  построению; мониторы убиваются атаками.

## БЛОКЕР #1 — решить до кампании

**Миссия заканчивается раньше атаки** (OPEN-1 в RESULTS_NOTES). uav_1 на
90-й секунде уже дома, маршрут пройден. Значит:
- mission resilience измерить не на чем,
- все текущие результаты — про атаку на **висящий** дрон.

Варианты: удлинить маршрут (`configs/experiment.yaml`, waypoints) или бить
раньше (но не раньше сходимости EKF, ~30-60с; 30с точно рано).
**Без решения кампания на 1160 прогонов соберёт не то.**

## Слабые места работы (знаем, лечим)

1. **Идеальный mesh** (ZeroMQ по localhost: 0 потерь, 0 задержки) при том,
   что Ch.2 разбирает FANET. Самое уязвимое. → пункт 5.
2. **Рой из 3 дронов** — «рій» с N=3 тонко. → масштабирование после.
3. **Rule-based детекторы vs Ch.2 про AI/ML** — разрыв обзор↔реализация.
   → явное объяснение в Ch.4 (детектор сменный), опционально ML-демо.
4. Только симуляция, нет реального противника → limitations.

## Сильные места (не растерять при написании)

- **Ground truth снаружи системы** — многие меряют последствия атаки

## Критические тех-факты (не переоткрывать)

- Param в полёте — ТОЛЬКО через mission MAVSDK System (ParamWriter Protocol).
  Второй mavsdk client не биндится на 14560+i.
- **4-й UDP endpoint в router ломает** MAVSDK PARAM_SET миссии. Держать 3.
- Мониторы пишут в JSONL только события → пустой monitor-JSONL на baseline
  это НОРМА.
- Params PX4 в `rootfs/{0,1,2}/parameters*.bson` — чистить между прогонами.
- gRPC-порты: mission 50051-53, loiter 50054-56.
- PX4-stdout в /dev/null по умолчанию (флаг `PX4_LOG=1`).
- Gazebo: `gz topic -e -t /world/default/pose/info --json-output`, 50 Гц,
  одно сообщение в строку, модели `x500_N` ↔ `uav_N`. Мир **ENU**
  (x=восток, y=север). `header.stamp` — **симуляционное** время, события в
  merged.jsonl — wall-clock; рекордер пишет оба.
- `SIM_GPS_OFF_N=50` → дрон физически уезжает ровно на 50 м **на юг**.
- Heartbeat-FP от нагрузки SITL — известный артефакт, бывает и до, и после
  атаки.
